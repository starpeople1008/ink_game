# AI로 저장소 받고 올리기 (초보자용 가이드)

git 명령어가 익숙하지 않아도 괜찮습니다. **AI(예: Claude Code, Cursor, Copilot, ChatGPT 등)에게 한국어로 부탁**하면 대신 해 줍니다. 이 문서는 "AI에게 뭐라고 말하면 되는지"를 알려줍니다.

> 핵심 원칙: **AI에게 "무엇을 하고 싶은지"를 말하고, AI가 보여주는 명령을 실행하기 전에 한 번 확인하세요.** 모르는 건 "이거 안전해?", "무슨 뜻이야?" 라고 다시 물어보면 됩니다.

---

## 0. 준비물 (한 번만)

새 컴퓨터라면 AI에게 이렇게 물어보세요:

> "이 컴퓨터에 Git이랑 Unity가 깔려 있는지 확인해 줘. 없으면 설치 방법 알려줘."

추가로 본인 정보를 등록해야 커밋에 이름이 남습니다:

> "Git에 내 이름이랑 이메일 등록해 줘. 이름은 OOO, 이메일은 OOO@example.com 이야."

---

## 1. 저장소 받기 (다운로드 = clone)

AI에게 이렇게 말하세요:

> "이 GitHub 저장소를 내 컴퓨터에 받아줘:
> https://github.com/starpeople1008/ink_game.git
> 홈 폴더 말고 D:\Projects 같은 작업 폴더에 받아줘."

AI가 `git clone` 명령을 실행해 줍니다. 끝나면:

> "받은 폴더를 Unity Hub에서 어떻게 여는지 알려줘."

> ⚠️ 받은 직후 `Library` 폴더가 없다고 당황하지 마세요. Unity에서 프로젝트를 처음 열면 자동으로 만들어집니다. **정상입니다.**

---

## 2. 작업 시작 전 — 최신 내용 받기

작업하기 전에 항상:

> "작업 시작할 거야. 원격에서 최신 내용 받아줘 (pull)."

이렇게 하면 다른 사람이 올린 변경 사항을 먼저 받아옵니다. (안 하면 나중에 충돌이 납니다.)

---

## 3. 내 작업 공간 만들기 — 브랜치

`main`에 바로 작업하지 말고, 내 작업용 브랜치를 만드는 게 안전합니다.

> "내 작업용 새 브랜치 만들어줘. 이름은 'feature/내가-하는-일' 로 해줘."

예: `feature/타이틀화면`, `fix/적-안움직임`

---

## 4. 작업한 내용 올리기 (업로드 = commit + push)

Unity에서 작업하고 저장한 뒤, AI에게:

> "지금까지 작업한 내용 전부 올려줘. 커밋 메시지는 '타이틀 화면 만듦' 으로 해줘."

그러면 AI가 `add → commit → push`를 알아서 해 줍니다.

올린 게 잘 됐는지 확인하고 싶으면:

> "방금 올린 거 GitHub에 잘 올라갔는지 확인해줘."

---

## 5. 내 작업을 main에 합치기

브랜치 작업이 끝났으면:

> "내 브랜치 작업 끝났어. main에 합치는 Pull Request(PR) 만드는 방법 알려줘."

보통 GitHub 사이트에서 버튼 몇 번으로 합칠 수 있고, AI가 단계별로 안내해 줍니다.

---

## 6. 문제가 생겼을 때 — AI에게 그대로 보여주기

빨간 글씨(에러)가 나오면 **그 내용을 그대로 복사해서** AI에게 붙여넣으세요:

> "이런 에러가 났어. 무슨 뜻이고 어떻게 해결해?  
> (에러 내용 붙여넣기)"

자주 겪는 상황과 AI에게 할 말:

| 상황 | AI에게 이렇게 말하기 |
|------|----------------------|
| push했더니 거부당함 (rejected) | "push가 거부됐어. 먼저 pull하고 다시 올려줘." |
| 충돌(conflict)이 났다고 함 | "충돌이 났대. 어떤 파일이고 어떻게 해결하는지 같이 봐줘." |
| 실수로 잘못 올림 | "방금 커밋을 되돌리고 싶어. 안전한 방법 알려줘." |
| 뭘 했는지 모르겠음 | "지금 내 저장소 상태(브랜치, 변경사항) 정리해서 알려줘." |

---

## 7. 안전 수칙 (꼭!)

- **`inking` 폴더 안에서만** 작업하라고 AI에게 미리 말해두세요. ("작업 폴더는 inking이야.")
- AI가 **삭제(`rm`, `delete`), 강제(`--force`, `reset --hard`)** 가 들어간 명령을 제안하면, 실행 전에 **"이거 위험한 거 아니야? 왜 필요해?"** 라고 꼭 한 번 되물으세요.
- 잘 모르겠으면 **실행하지 말고** 물어보세요. git은 대부분 되돌릴 수 있지만, 강제 명령은 되돌리기 어렵습니다.
- 정기적으로 작업물을 push 해두면, 컴퓨터에 문제가 생겨도 GitHub에 안전하게 보관됩니다.

---

## 한눈에 보는 흐름

```
[받기]   AI: "이 저장소 받아줘 (주소)"        → clone
   ↓
[시작]   AI: "최신 내용 받아줘"               → pull
   ↓
[작업]   AI: "내 작업용 브랜치 만들어줘"        → switch -c
   ↓     (Unity에서 작업 + 저장)
   ↓
[올리기] AI: "작업한 거 올려줘 (메시지)"        → add + commit + push
   ↓
[합치기] AI: "main에 합치는 PR 만들어줘"        → Pull Request
```

자세한 명령어 설명은 같은 폴더의 `GIT_GUIDE.md`를 참고하세요.
